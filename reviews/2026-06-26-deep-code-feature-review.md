**Date:** 2026-06-26 · **Scope:** `index.html` (PWA, ~5,860 lines), `pipeline/` (ingestion app, bookmarklets, Apps Script), supporting docs.

## How this review was produced (and how much to trust it)
A 7-subsystem deep read (map/sync, data/cache, prediction/analytics, issue-correction, bottles/photos/themes, pipeline app, bookmarklets+backend), then **every stability/correctness claim was adversarially re-verified against the actual code**: 30 bug claims → **28 confirmed/partly-confirmed, 2 refuted**, with several severities honestly *downgraded* on verification. ## The big picture (read this first)

Your instincts as a builder are good. Almost every confirmed defect traces back to **two root causes**, not dozens of unrelated mistakes:

1. **State that must stay in sync is synced by hand.** The dual-map architecture requires every mutation to update *both* Leaflet maps; there's no single function that does it, so two desktop entry points already forgot. Same smell powers the shared-`currentZoom` glitch and the mobile/desktop copy-paste twins.
2. **Timestamps are written inconsistently and trusted blindly.** App-logged drops are stored in **UTC** wall-clock; pipeline drops are stored in **Eastern**; analytics reads both as local. Your single `timestamp` column holds two different clocks — and that column *is* the prediction signal.

And one **strategic gap** above all the bugs: **the app summarizes the past per-district; it does not yet predict per-store.** It never actually answers your two stated core questions — "which store drops *next*" and "which store gets the *best* bottle." The data to do both already exists and is partly parsed-then-discarded. Closing that gap (Q6: 6a/6b/6c) is the single highest-value direction for the product.

## Q1 — Changes / things I'd do differently (map + pipeline)

### Map part of the app
- **1a. Collapse the two maps behind one `refreshAll()` instead of remembering to update both by hand.** (M) "Update both maps" is a rule in `CLAUDE.md` enforced by memory, and it already broke twice: desktop Quick Drop (`quickDrop` `index.html:4350`) and the desktop modal save (`confirmLog` ends `closeModal();refresh();` `index.html:2852`) both call `refresh()` (`index.html:2282`), which only touches the mobile map. Don't add more wrappers — make one function the only way to refresh: `if(map) renderMarkers(); if(desktopMap){ renderDesktopMarkers(); updateDesktopStats(); buildSbDistricts(); }`. The architecture then *stops allowing* the bug. Highest leverage-per-effort change in the app.
- **1b. Make pin rendering per-map, not driven by shared globals.** (S) `currentZoom`/`currentPinStage` are single globals (`index.html:1381-1382`) written by both maps' `zoomend` handlers, and `makePin` reads the global rather than the zoom of the map it's drawing (`index.html:1949`). Once both maps are alive, zooming one can render the other's pins at the wrong stage. Pass the target map's zoom into `makePin`.
- **1c. Switch from full-teardown rendering to incremental marker updates.** (M) Every log/mark/wave-switch/color-cycle destroys and rebuilds *every* marker (`renderMarkers` does `forEach(removeLayer); markers={}` then recreates all — `index.html:2196`). That's why open popups slam shut mid-read. Add `updateMarker(storeId)` using `marker.setIcon(...)`; reserve full rebuilds for filter/color/zoom-stage changes. This also unlocks cheap "go here next" pin pulses (1e).
- **1d. Decide what the desktop map *is* and stop half-supporting field features on it.** (S) `desktopYouMarker` is placed once and never updated by the geolocation watch (`index.html:3496`, `1900-1903`), implying a live position that isn't. `CLAUDE.md` says GPS/distance is mobile-only by design — so make the code match: drop the stale desktop "You" dot.
- **1e. Turn the map into the prediction surface, not just a status board.** (M, after 1c) `makePin` already branches on `colorMode` (district/tier/off, `index.html:1950`). Add a 4th mode that colors/badges pins by **predicted next-drop likelihood** and pulses the most-likely-next + best-bottle stores. Today the map answers "what happened"; your stated reason to exist is "what happens next."

### Pipeline process
- **1f. Stop reconstructing the channel↔thread join from a timestamp — carry the Discord message id end-to-end.** (M) `dropId = store + "_" + canonTS(msg_ts)` (`pipeline/index.html:590`) only works under the assumption (`pipeline/index.html:334`) that the channel `!drop` and the thread root are the same message in the same Eastern minute. But `autoParseThread` derives the thread time from `rows[0]` — the *earliest captured* message (`pipeline/index.html:823`) — so if the starter isn't in the captured DOM, the thread's bottles silently attach to a *new* drop row. You already capture the immutable `raw_id` and dedup on it — key the join off that, not a reconstructed minute. This is the single most fragile assumption in the pipeline, and it fails *invisibly* (the worklist matches on store, so a split join shows "thread captured").
- **1g. Make `canonTS`/`estParts` idempotent and stop laundering timestamps through browser-local time.** (S) `canonTS` does `new Date(ts)` then re-expresses in `America/New_York` (`pipeline/index.html:337`) — correct for Discord's offset ISO, but for an already-canonical offset-less string it parses in the *browser's* zone first. Pick one canonical format (Eastern wall-clock, no offset), generate it identically everywhere, and make the normalizer a no-op when input already matches. A 4–5h skew in hour/day stats is the worst kind of bug for a prediction app: it never crashes, it just quietly makes the answer wrong.
- **1h. Validate parsed store numbers against the Stores reference before they become data.** (S) `firstStoreNum` (`pipeline/index.html:567`) grabs the first 2–3 digit run anywhere in freeform text, so "nada, last 107 gone" → store 107 and "still some 22 left" (HH22) → store 22. These become `checks` sightings that feed nada *bracketing* — phantom dry stores corrupt your negative-evidence signal. `knownStore()` already exists (`pipeline/index.html:330`); gate the fallback on it.
- **1i. Give `drop_bottles` a real dedup key — don't append it keyless.** (S) `Drops` upserts on `dropId`, `lexicon` on `raw_term`, but `drop_bottles` is written with no key (`pipeline/index.html:757` → `appendObjects_(sheet, objs, null)`). Your UI says "Click Save again to resume" — only safe for keyed writes. Use the composite `dropId + '#' + cellar_id` you already compute elsewhere. (Real but low-frequency: only fires on a save failure.)
- **1j. Chunk the Cellar resolve URL the way you already chunk writes.** (S) `writeRows` splits GETs at ~1800 chars (`pipeline/index.html:264`), but `resolveTokens` jams the whole token list into one GET (`pipeline/index.html:237`). `parseChannel`'s `allToks` is unbounded; on overflow, tokens silently fail to resolve and their bottles fall through as `unmatched` with no log of which. Reuse the chunking you already wrote.
- **1k. Add a server-side capture-health/schema-check endpoint.** (M) The backend silently drops any object key that isn't already a column header (it bit you once with `msg_link`) and returns `success:true` anyway. A tiny `doGet?action=health` returning per-tab row counts, oldest/newest timestamps, and missing-required-headers turns "discover the dropped column weeks into the season" into "see it before the season starts."

**Through-line:** your good patterns (manual dual-map for control, raw-capture-then-parse, `raw_id` dedup, chunked writes) failed only where they weren't applied *consistently*. The durable win in each half is to make the correct path the *only* path: one `refreshAll()` for the map; one canonical timestamp + one `raw_id` join for the pipeline.

## Q2 — Top 5 stability + Top 5 efficiency improvements

### Stability (ranked by real payoff)
- **2a. Fix the timezone bug that silently poisons every "when does it drop" prediction.** *(high — confirmed)* `_quickMarkCore` (`index.html:2785`), `quickDrop` (`4354`), and the modal default `_openModal` (`2809`) stamp time as `new Date().toISOString().slice(0,16)` = **UTC**. Analytics reads offset-less strings as **local** (`computeDistrictAnalytics` hour bucket `4114`, day-of-week `4129`), a 4–5h skew; late drops roll to the wrong weekday. Meanwhile pipeline drops are correct Eastern → two clocks in one column. Fix: one `nowLocalISO()` helper (build from local parts like the pipeline does) used at `2785/4354/2809` (and the modal's derived `dayOfWeek` at `3368`). **This is the most damaging confirmed defect in the codebase.**
- **2b. Stop the desktop "Fill In Details" flow from creating duplicate drop rows.** *(high — confirmed)* `fillInQuickDrop` stashes the queue id (`index.html:4434`) but `efSubmit` never reads it: it builds a fresh row with a different column layout and no `dropId` (`3375-3402`) and calls `addDrop` again (`3408`) → a **second** sheet row for the same drop, and never clears the queue item. Duplicate `drop` rows inflate per-store frequency — the exact input to "which store drops next," biased toward stores you touch most. Fix: read `btn.dataset.quickDropId`, carry the original `dropId` so the backend updates instead of appends, clear `quickDropItems` + matching `pendingDrops`.
- **2c. Guard the localStorage parses so one corrupt key can't brick the whole app.** *(medium — confirmed; blast radius justifies medium over the original "low")* `pendingDrops` (`index.html:1378`) and `quickDropItems` (`4348`) load with bare `JSON.parse` at top-level script execution; `||'[]'` handles a *missing* key, not a *truncated* one (real on iOS Safari mid-write/quota eviction). A throw aborts the whole inline script → no functions defined, `initAndDetect()` never runs → blank app **and** your unsynced field drops are unrecoverable. `loadFromCache` (`1867-1872`), `issueCorrections` (`2479`), `districtBehaviors` (`4846`) share it. Fix: one `safeParse(key, def)` everywhere.
- **2d. Make field corrections actually reach the sheet (and fix the broken review wave dropdown).** *(high — confirmed)* Two coupled defects: (1) every correction path (`resolveCollision`, `saveIssueEdit/BotEdit/ReporterEdit/LinkEdit`, `saveReviewDrop`, `saveFillin` — `index.html:2678/2695/2709/2723/2737/4551/4762/4825`) updates local cache then fires `scriptPost('updateDrop',…)` fire-and-forget with the error swallowed; the `issueCorrections` outbox is **written but never read** (grep-confirmed); next `loadFromSheet` overwrites `DROPS` so the fix silently reverts. (2) `renderReviewList` filters waves with a double-escaped `/^\\d{4}/` (`index.html:4483`) — matches a literal backslash, returns `[]` — while line `4475` in the *same function* uses the correct `/^\d{4}/`. The wave `<select>` is therefore always empty, so the most important review action (assign a wave) is dead. Fix: the one-char regex fix + a real `applyCorrection()`/`flushCorrections()` replayed wherever `flushPending()` runs. **Caveat to verify yourself:** `updateDrop` isn't in the *pipeline* `Code.gs` (the main app's Apps Script isn't in this repo) — confirm your deployed main-app script actually implements `updateDrop`, or these corrections never persist regardless of network.
- **2e. Escape scraped text before it hits the DOM (stored-XSS surface).** *(high — confirmed; mitigated by single-user use)* No escaping helper exists anywhere (grep-confirmed). Worst sink: `buildBtUnknownList` `index.html:4986` → `onclick="saveBtIdentify(${i},'${t.raw}','${p}')"` — a quote or `</script>` in a scraped token breaks out and executes; edited names persist to the sheet → *stored* XSS. `bottleChipsHTML` (`2334`) fires on ordinary drop rendering. Fix: `escapeHtml()`/`escapeAttr()` applied everywhere, and pass only ids (never free text) into inline handlers.

> Honorable mention: the desktop quick-drop/modal-save stale-pin issue (2a's cousin) is real but self-healing and desktop-only (medium) — the `refreshAll()` fix in 1a kills the whole class.

### Efficiency (ranked by real payoff)
- **2f. Reload only the tabs you changed (pipeline) — the dominant latency cost.** Every mutation calls `loadAll()` (`pipeline/index.html:388`) = GET all 8 tabs. Confirming one wave (`953`) re-pulls `raw_channel`+`raw_threads` (your biggest tables) though only `Waves`/`Drops` changed. Scales against you as you backfill. Fix: `loadAll(['Waves','Drops'])` or splice known updates into `DATA` in memory.
- **2g. Stop rebuilding every map marker on every log.** Both renderers tear down all markers and recreate every store's pin on every `refresh()` (`index.html:2196`/`3507`). On a phone that's hundreds of `divIcon` creations per log + an open popup that closes. Fix: incremental `updateMarker(storeId)` (see 1c).
- **2h. Read one column, not the whole sheet, on each pipeline write (the backfill ceiling).** Each `doPost` batch does `appendObjects_` → `readTab_` → `getDataRange().getValues()` over the *entire* growing tab to rebuild its dedup set (`pipeline/appsscript/Code.gs:135`). A 12k-message backfill in 400-row batches ≈ O(n²) cells read — the practical wall on how deep you can backfill before the 6-min limit. Fix: read only the dedup column, or POST the whole paste in one body.
- **2i. Memoize district analytics instead of re-scanning all drops ~7× per filter change.** `computeDistrictAnalytics` (`index.html:4066`) does a full `allDrops().filter` + several O(n) passes; `setAnalyticsFilter` (`4063`) re-renders *all* districts when one changed. Unlike `getStoreStatus`/`getDropOrder` (memoized at `1400-1435`), analytics has no cache. Fix: memoize per `(district, filter-signature)` keyed off the `allDrops` cache generation; re-render only the changed card.
- **2j. Cache unknown-bottle tokenization instead of re-tokenizing all history on every keystroke.** `getUnknownBottleTokens` (`4895`) iterates `allDrops()` + `normalizeBottlesText` (prefix-match over all lookup keys); `filterBottlePane` (`5181`) calls the builders on *every keystroke* → re-tokenizes your whole history twice per character. Fix: cache keyed by drop-cache version, debounce the search, filter the built list in place.

## Q3 — New self error / issue-correction features

**What already exists (not re-proposed):** a 5-type issues queue (`buildIssues` `index.html:2483`: OF1924 collision, needs-review, missing bottles/reporter/link), a needs-review queue, an idle fill-in task queue, the OF1924 resolver, and an `issueCorrections` log (`2479`). New, additive ideas below:

- **3a. Timezone-skew sanitizer (highest value).** Detector: flag drops whose local hour is 1–7 AM (almost certainly UTC-stored app logs) as a new "Time looks wrong" issue with one-tap "shift −4h/−5h." Cross-source: pipeline drops carry `minutesSinceMidnight`/`dayOfWeek` *and* a raw `timestamp` (`index.html:1710`); if they disagree by a near-constant 4–5h, flag and exclude from stats until fixed. Root fix = 2a.
- **3b. Duplicate / conflicting drop detection (new issue type).** Nothing in `buildIssues` looks for real duplicate *rows* — it only de-dups its own issue IDs. Group `allDrops()` by `storeId+wave`; flag (1) >1 `drop` in a group (possible duplicate → keep-one/merge) and (2) a `drop` + `no_drop` in the same group (conflict → force a pick). Reuses the existing inline-resolve UI; protects the frequency signal behind "most likely to drop next."
- **3c. Channel-vs-thread reconciliation alert (consumer-side audit).** Detect the failed-join signature in the data the PWA reads: a store with captured thread bottles but a `drop` row with empty bottles, plus a *second* `drop` for the same store within minutes → "Split drop — bottles on wrong row" with a merge action. (Durable fix is upstream = 1f, but this catches whatever the pipeline misses.)
- **3d. Reporter trust / confidence scoring.** `reporter` (`index.html:1706`) is only ever displayed. Compute per-reporter corroboration rate (backed by another reporter/thread vs solo), contradiction rate (later overturned), and a volume floor (`n=`). Surface a confidence dot in the queues; optionally auto-route solo reports from never-seen reporters into needs-review. Lets the predictor weight hand-verified drops above raw scrapes — no new data needed.
- **3e. "Did my correction actually stick?" reconciliation on reload.** Turn `issueCorrections` into a real outbox (`synced:false`, *all* handlers route through it — today three edit handlers leave no record), add `flushCorrections()` on load/reconnect, and after `loadFromSheet` **diff** fresh sheet rows against recently-saved corrections by `dropId`; mismatch → "Correction didn't take" instead of silently re-flagging. Converts the whole self-correction layer from optimistic-and-silently-failing to guaranteed-with-an-alarm.
- **3f. Sample-size confidence flags on every prediction output.** "Predictability %" is just top-mode % with no `n` guard — 2 waves trivially reads 100%/"HIGH"; an empty district even renders a fabricated "Sun 0%." Gate "HIGH"/"MEDIUM" on a floor (e.g. ≥5 waves), show `n=` explicitly, shrink the raw % toward 50% when `n` is small (Wilson/Bayesian), render "—" on zero buckets. A correctness-*of-meaning* fix: stop sounding confident on almost no data.
- **3g. Unknown-bottle-token health meter (vocabulary self-healing).** Unmatched scraped tokens fall silently out of all tier/quality analysis. `getUnknownBottleTokens` already aggregates them with counts + which stores; surface a banner ("14 unidentified tokens"), sort triage by frequency × tier-guess, and prompt inline during logging ("Is `XYZ` = `<closest match>`?") so the vocabulary self-heals as you log.

**Sequencing:** 3a first (it corrupts the stats everything else depends on); then 3b + 3f (mostly additive); then 3e (makes all cleanup actually reach the predictor); then 3c/3d/3g.

## Q4 — Weakest points / sloppiest code (worst-first)

- **4a. The single 5,860-line `index.html` with one giant inline `<script>` (1338–5861) — the systemic root cause.** Everything shares one room with no walls. It's *why* a corrupt localStorage key can brick the app (top-level unguarded `JSON.parse` at `1378`/`4348` runs during parse; a throw means `init`/`buildMap`/`initAndDetect()` never define), *why* "remember to call the other function" is the only safety mechanism, and *why* globals act as implicit parameters (`currentZoom`/`currentPinStage` `1381`, `btPrefix` `1376`, `window._reviewDrops` `4487`). Even splitting into a few `<script type="module">` files removes most of this report's class of bug.
- **4b. Dual-map sync as a manual discipline — the most bug-dense pattern in the app.** `refresh()` (`2282`) is entirely mobile-only; some entry points hand-wrote desktop wrappers (`syncFromSheet`, `quickMark` `4025/4031`), two didn't (`quickDrop` `4350`, `confirmLog` `2852`) → two confirmed stale-pin bugs with one root cause. Plus the shared-`currentZoom` cross-map glitch. Fix is structural: one `refreshAll()`.
- **4c. Fire-and-forget writes with no escaping, no retry, and a write-only "outbox."** For an app whose value is the integrity of the drop table, this is the sloppiest layer: corrections lost offline with no retry (and `resolveCollision`/`saveFillin`/`saveDesktopFillin` toast a *false* "Saved ✓" on failure); `issueCorrections` written in 2 places, read in 0; wrong-row corruption via non-unique `storeId+wave` match keys (`2689/2705/2719/2733`) when a unique `dropId` is parsed at `1713` and ignored locally. One `applyDropCorrection(dropId, …)` keyed on `dropId` through a replayable outbox collapses three issues at once.
- **4d. Regex parsing of human-typed Discord text — fragile by nature, and it leaks into prediction inputs.** Bare-number store fallback misreads proofs/ages as stores (`firstStoreNum` `pipeline:567`, no review flag → corrupts nada bracketing); 2-char shortcodes matched by *substring* (`findHit` `pipeline:611`, `detectStatus` `807`) so `ER` matches `Weller`; `extractCandidates` silently `.slice(0,80)` (`794`) drops the latest-mentioned bottles. Deeper: **four near-identical token-resolution blocks** (`findHit` 611, `resolveThread` 712, `autoParseThread/normMatch` 836, `reResolveParsed` 879) have *drifted to different matching logic* — same token resolves in one flow, not another.
- **4e. Pervasive unescaped interpolation → stored-XSS surface (confirmed high).** See 2e. Today's escaping is ad-hoc per-call (`btMergeSearch` escapes only `'`, `buildBtKnownList` only `"`, most sites nothing) — the inconsistency *is* the root cause.
- **4f. The picker crashes on every locally-created bottle (confirmed high).** Local bottles get a **string** id (`'local_'+Date.now()` `5141`) but the picker emits it *unquoted* into the handler (`onmousedown="pickerAdd('${ctx}',${b.id})"`) → `pickerAdd('mob',local_1719…)` is an undeclared identifier → `ReferenceError`, bottle permanently unselectable; and `pickerAdd` compares `b.id===bottleId` strictly (`2393`). The rest of the file correctly uses `String(b.id)===String(id)` (`5191/5162/4926`) — the picker is the one place that didn't get the memo.
- **4g. Copy-paste duplication that has already drifted.** `buildMap`/`buildDesktopMap`, `renderMarkers`/`renderDesktopMarkers`, `renderIssuesList`/`renderDpIssuesList`, the three bookmarklets' `toast`/`copyText`/`channelName`. Already diverged into latent bugs: desktop bottle-edit pre-fills `value=d.bottles` (`3101`) while mobile doesn't (`2597`); `normMatch` drops `tier` while `resolveThread` drops `shortcodes` from the *same* Cellar response. Every fix must be applied 2–3× or it drifts further.
- **4h. Local offenders worth fixing on sight:** the double-escaped review-wave regex (`4483`, dead dropdown); UTC-vs-local timestamp trio (`2785/4354/2809`); `updateZoomBadge(z)` ignores `z` and is never called (`1922`, dead code with a lying signature); `window._reviewDrops` keyed by render index with a silent `slice(0,50)` cap (`4487/4489` — key by `dropId`); `btPrefix` mutable global used as an implicit parameter (`1376`, set/reset in `buildMobileBottles` `4939`).

## Q5 — Make code more efficient WITHOUT losing features (behavior-preserving)

Dominant theme: **mobile/desktop copy-paste twins** + write-path duplication. Each item below changes *how* code is organized, not *what the app does*.

- **5a. Collapse `refresh()` + the two map renderers into one `renderMarkersOn(targetMap, store, onClick)` + a `refreshAll()`.** The renderers are ~90% identical; only the per-marker click wiring differs (`bindPopup` vs `on('click')`). Removes ~25 duplicated lines and the entire "updated one renderer, forgot the other" class (and the two confirmed stale-pin bugs as a free side effect).
- **5b. Merge `buildMap()` + `buildDesktopMap()` into one `buildMapInstance(elementId, {onZoomEnd})`.** Near-verbatim copies (same Leaflet opts, Voyager tile URL, zoom control, `you-dot`, `zoomend` logic). The documented tile-layer changes currently must be edited in two places. ~12 lines.
- **5c. Delete the dead `updateZoomBadge()` (`index.html:1922`).** Takes a `z` it never reads, never called from anywhere (confirmed). Removes misleading dead code.
- **5d. Share one `issueCardHTML(issue, idPrefix)` between `renderIssuesList` (`2548`) and `renderDpIssuesList` (`3053`).** ~90% identical, ~80 duplicated lines → one template; also removes the unintended mobile/desktop pre-fill inconsistency.
- **5e. Add `escapeHtml()`/`escapeAttr()` and route all interpolations through them.** Replaces a scattering of partial hand-rolled `.replace()` calls (`5063` only `"`, `5009` only `'`, most nothing) with two named helpers — *fewer* characters overall, one place to reason about, and it closes the XSS class.
- **5f. Standardize id comparisons on `String(a)===String(b)`.** Picker uses strict numeric `b.id===bottleId` (`2393/2400/2418`) while the rest uses `String()===String()` (`5117/5162/5191`); local bottles get string ids → the two conventions disagree (root cause of 4f).
- **5g. Build a `storesById` index once instead of repeated `STORES.find`/`forEach` scans.** `applyDistrictDimming` (`5723`) rebuilds a store→district map on *every* district-chip tap; the pipeline re-scans `DATA.Stores.find` per drop (`pipeline:328`). `STORES` changes only on load.
- **5h. Consolidate the wave-label parsing scattered across 4+ functions** (`computeDistrictAnalytics` `4136`, `buildSbWaveHistory` `5275`, `renderMobileWaves` `5373`, `autoDetectWave` `1771`) into one `parseWaveDate(label)` + a `waveIndex` built once per load. Also kills the *fifth* copy's typo (the `/^\\d{4}/` dead dropdown).
- **5i. Pass `btPrefix` as an argument instead of mutating a global** (`1376`, reset dance in `buildMobileBottles` `4939`). The signature then tells the truth; no reset to forget.
- **5j. Extract one `toSheetRow(drop)` + one `buildBottlePayload(b)`.** Two divergent drop→row serializers target the same `addDrop` (`writeDropToSheet` 17 cols `2768` vs `efSubmit` ~26 cols `3375`); the bottle payload is assembled 3× with drifting shapes (`5122/5165/5142`). A column change becomes a one-line edit (this is the structural fix under the 2b duplicate-drop bug).

## Q6 — 10 new features (mix of impact / cool / QoL)

Deliberately distinct from Q1–Q5 and from the `TODO.md`/`FEATURES.md` backlog. Tags: `[prediction]` `[cool]` `[QoL]` `[data]`.

- **6a. Live "Drop Radar" ranked target board `[prediction]`.** One always-on panel ranking every store by a computed **next-drop probability** for the active wave (its own cadence = avg days between *its* drops, days-since-last, day-of-week concentration, district overdue status), re-sorting live as drops land. This is the headline missing piece: today nothing ranks individual *stores* by likelihood. Data exists (`getStoreHistory`); only the per-store model is missing. Turns analytics from rear-view mirror into hunting GPS — the #1 goal.
- **6b. "Best Bottle Heat" prediction layer `[prediction]`.** Per-store quality score from the already-parsed-but-unused `topBottle`/`dropQuality`: "gets the wave's top bottle 38% of the time · avg quality A−," surfaced as a badge and a 4th map color mode. Answers your second core question ("which store gets the BEST bottle") from data currently parsed and thrown away. Zero new collection.
- **6c. "In-Wave Cluster Forecast" live timer `[prediction]`.** For multi-cluster districts, the moment the first drop is detected: "Second cluster historically fires ~3.5h later → expected ~1:30 PM," using the `clusterA`/`clusterB` timing fields the pipeline computes and the PWA parses but never reads. Most useful at the exact high-stakes moment — mid-wave, deciding whether to keep hunting.
- **6d. Confidence-graded predictions ("n=3, low confidence") `[prediction]`.** Every prediction number gets a sample-size badge and is shrunk toward neutral on thin data (`HIGH · n=12` vs `LOW DATA · n=2`). Prerequisite for actually acting on predictions. (Productizes 3f.)
- **6e. "Drop Replay" animated cycle timeline `[cool]`.** A play button that animates a whole wave on the map — pins light up in real chronological drop order, with scrubber + speed. Makes intra-wave sequencing ("who drops first") viscerally obvious, exposes outlier waves, and trains your own intuition. (Upgrades the flat "timeline view" backlog idea into something spatial.)
